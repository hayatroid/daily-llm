# /new 🗣️

## 目的

トピックについて対話し、会話内容を「思考の実演」として構造化された価値ある記事に変換する

## 使い方

`/new [トピック]`

## 根本問題の認識 🎯

### なぜ多くの記事は価値を失うのか

現代の情報環境では「まとめ記事」が量産されている：

- 対話の要素を抽出し、構造化した結果を羅列
- 読者が楽に消費できる「分かりやすい」形式
- 即座に結論に到達できる効率的な構成

しかし、このアプローチには根本的な問題がある：

**読者の思考プロセスを奪ってしまう**

真に価値のある学習は、結論を与えられることではなく、**自分で洞察に到達するプロセス**にある。「分かりやすい」記事は、この最も重要な部分を奪ってしまう。

### 「思考の実演」という解決策

対象：読者が同じ発見プロセスを追体験できる記事構造

**核心原理**：

- 洞察の「結果」ではなく「プロセス」を伝える
- 読者が自ら「気づき」に到達できるよう導く
- 記事自体が学習体験として機能する

これは決して「難しくする」ことではない。むしろ**本質的な理解を促す**ための設計である。

## 実行手順 📋

### Phase 1: 対話 (Deep Dialog)

1. **トピック確認**: 提供されたトピックについて対話開始

2. **深掘り対話**:

   - **重要**: すべての対話で「think harder」を実践
   - 表面的な理解で満足せず、より深い層まで探求
   - 「なぜそうなるのか」「本質的な理由は何か」を追求
   - 5-10 往復の対話完了（深さに応じて調整）
   - 一つの洞察から別の洞察へと自然に展開するまで継続

3. **洞察の連鎖確認**:

   - 対話が論理的必然性のある流れになっているか確認
   - 表面的な話題の羅列ではなく、深い理解に到達したか確認

4. **文書化提案**: 十分な思考プロセスが蓄積された時点で記録を提案

### Phase 2: 思考プロセスの再構成 (Thought Process Reconstruction)

**❌ 従来アプローチ（要素抽出）は使わない**：

- 対話から「要素」を抽出して整理
- 標準的な構成パターンに当てはめ
- 結論を効率的に伝える構造

**✅ 思考プロセスの再現**：

1. **対話ログの分析**

   - 対話ログを時系列で読み返す
   - 「あ、ここで理解が深まった」瞬間を特定
   - 「この発言がなぜ次の発言を生んだか」の因果関係を探る
   - 専門概念が「必要に迫られて」登場した文脈をマーク

2. **思考の流れの抽出**

   - 最初の問題意識は何だったか？
   - どんな順序で理解が深まったか？
   - どの段階で「大きな気づき」があったか？
   - 最終的にどんな統合的理解に到達したか？

3. **読者体験の設計**
   - 読者が最初に持ちそうな疑問から始める構成
   - 一つずつ理解を積み重ねられる順序
   - 「ここで気づくはず」のタイミングで概念を導入
   - 読者が「発見した！」と感じられる瞬間を作る

### Phase 3: タイトル協議

**think harderでタイトルを決定**：

- 表面的な要約ではなく、思考の本質を表現
- 8-30 文字で記憶に残るフレーズ
- 読者の好奇心を喚起する表現
- ユーザーとの協議で最適解を見つける

### Phase 4: 記事執筆 (Thoughtful Article Creation)

1. **メタデータ生成**:

   - **tags**:
     - **タグ統計の活用** (tag-stats コマンドの原理を適用):
       - `/tag-stats` を実行して既存タグの統計情報を確認
       - **活性タグ優先**: 過去 3 ヶ月以内に使用されたタグから選択
       - **類似タグの統合**: 似た概念のタグは統一（例: dialogue/dialog-driven → dialogue）
       - **新規タグは慎重に**: 既存タグで表現できない概念のみ新規作成
     - **第 1 タグ**: 必ず `#life`（思想寄り）か `#develop`（技術寄り）を選択
     - **その他のタグ**: 思考プロセスで重要だった概念（全体で 3-5 個）
       - 高頻度タグ（5 回以上使用）から優先的に選択
       - 中頻度タグ（3-4 回使用）も積極的に活用
       - 単発タグ（1 回のみ）は類似概念がないか再検討
     - **形式**: 必ず slug 形式（小文字、ハイフン区切り）を使用
     - **注意**: `gem` タグは付与しない（ユーザーのみが判断）
   - **description**: 思考の流れの本質を 1 文で表現

2. **執筆における「think harder」の実践**:

   **✅ 必須原則**:

   - **自然な導入**: 一般的な問題意識や違和感から開始
   - **発見の表現**:「気づく」「見えてくる」「思い浮かぶ」など思考の動きを表現
   - **必然的展開**: 前の洞察が次の洞察を自然に導く構造
   - **文脈的導入**: 専門概念を思考の流れの中で自然に登場させる
   - **読者の能動性**: 結論を押し付けず、読者が自ら気づけるよう導く

   **❌ 避けるべき要素**:

   - いきなりの具体例や専門用語
   - 要素の機械的羅列
   - 結論先行の効率的説明
   - 受動的消費を促す「親切すぎる」構造

   **語り口**:

   - AI 視点でのユーザ観察記録として記述
   - 思考プロセスの実演として構成
   - 読者が同じ洞察プロセスを追体験できるガイドとして機能

   **数式記法の活用**:

   - **インライン数式**: `$...$` 形式を使用（例: $E = mc^2$）
   - **ブロック数式**: `$$\n...\n$$` 形式を使用（開始 $$ と終了 $$ の前後に改行）
   - **記事内容の種類に応じた活用**:
     - 数学的概念を扱う記事では積極的に数式を使用
     - 統計や確率の議論では可視化のため数式を活用
     - アルゴリズムの計算量などでも適切に数式表現
   - **読者体験の向上**: 数式により概念の正確な理解を促進

3. **構造の具体例**:

```markdown
## 違和感の芽生え

[一般的な状況への疑問から始める]

## 問題の核心

[違和感を具体化していくプロセス]

## 深層への到達

[表面的理解を超えて本質に迫る]

## 洞察の展開

[一つの理解から関連する理解へ]

## 統合的理解

[複数の洞察を統合した深い理解]
```

4. **ファイル作成**:

   - 保存先: `src/content/daily/YYYY-MM-DD/NNN-slug.md`
   - 品質チェック（format, lint, textlint）は不要（huskyがコミット時に自動実行）

5. **日次サマリー自動更新**:

   - **重要**: 記事作成完了後、自動的にその日のサマリーを更新
   - `/summarize [今日の日付]` を内部実行
   - 既存 index.md は参照せず、その日の全記事から完全新規生成
   - 新しい記事を含めた知的探求プロセス全体を再分析
   - 手動でのサマリー作成は不要（ダミーサマリーシステムを廃止）

## 成功の判断基準 🎯

### 対話の深さ

- ✅ 表面的な理解を超えて根本原理に到達
- ✅ 複数の「なぜ？」を経た思考の深化
- ✅ 一つの洞察から関連洞察への自然な展開
- ✅ 専門概念の本質的理解
- ✅ 読者が追体験できる豊富な思考プロセス

### 記事の機能

- ✅ 思考の必然的流れが明確に再現されている
- ✅ 読者が同じ発見プロセスを追体験できる
- ✅ 記事自体が学習体験として機能している
- ✅ 専門概念が文脈の中で自然に理解できる
- ✅ 受動的消費ではなく能動的思考を促進している
- ✅ 後から読み返しても思考プロセスを再体験できる

## 記事構成の原理 📄

### 基本的な流れ

```markdown
[ユーザ]は[自然な問題意識・違和感]から対話を始めた。

[導入: 一般的状況への疑問]

## [第一の気づき]

[問題意識の具体化プロセス]
[思考の動きを表現: 「見えてきた」「気づいた」等]

## [深化する理解]

[前の洞察から必然的に生まれる次の理解]
[具体例の自然な導入]

## [本質への到達]

[専門概念の文脈的登場]
[思考の流れ: 「ここで重要な概念が思い浮かんだ」等]

## [洞察の統合]

[複数の理解の統合と深化]
[より根本的な原理の発見]

## [メタ認知]

[思考プロセス自体への気づき]
[学習体験の意味づけ]
```

### 重要な技法

1. **問題意識の自然な芽生え**

   - 「何か違和感があった」
   - 「腑に落ちないものがあった」
   - 具体的状況から一般的疑問へ

2. **思考の動きの表現**

   - 「考えを深めていくと」
   - 「さらに深い層に向かった」
   - 「ここで重要な気づきがあった」

3. **必然的な概念導入**
   - 「この時、ある概念が思い浮かんだ」
   - 「改めて考えてみると」
   - 思考の流れの中での自然な登場

## 実行例 💫

```bash
# 入力例
/new "効率化ツールの弊害"

# 対話プロセス
1. 表面的議論: 「時間節約になる」
2. think harder: 「本当に価値を生んでいるか？」
3. より深く: 「思考プロセスへの影響は？」
4. 本質到達: 「判断能力そのものが劣化している」
5. 関連洞察: 「教育・メディアでも同じ構造」
6. 統合理解: 「便利さと真の価値の関係性」

# 記事出力
- 一般的な便利さへの疑問から開始
- 段階的な気づきのプロセス
- 関連分野への洞察の展開
- 読者が同じ思考を追体験できる構造
```

## よくある失敗パターン ❌

### 対話段階での失敗

- **表面的な議論で満足**：「便利ですね」「そうですね」で終わる
- **一問一答の羅列**：深い思考の連鎖が生まれない
- **専門用語の早期導入**：文脈なしに概念を説明し始める

### 構造化段階での失敗

- **時系列のコピー**：対話の順序をそのまま記事にする
- **要素の機械的抽出**：「ポイント 1、ポイント 2」の羅列
- **結論先行**：最初に答えを示してしまう

### 執筆段階での失敗

- **いきなり具体例**：読者の問題意識を無視した開始
- **説明的な文体**：「〜です」「〜します」の連続
- **飛躍のある展開**：読者を置き去りにした論理の跳躍

## 記事完成時チェックリスト ✅

### 構造面

- [ ] 一般的な問題意識から始まっているか？
- [ ] 各セクションが前のセクションから必然的に導かれているか？
- [ ] 専門概念が文脈の中で自然に登場しているか？
- [ ] 読者が「発見」できる瞬間が複数あるか？

### 文体面

- [ ] 思考の動きを表現する語句（「気づいた」「見えてきた」等）を使っているか？
- [ ] 読者に問いかける形式を適切に使っているか？
- [ ] 説明的でなく体験的な文体になっているか？

### 全体面

- [ ] 最後まで読みたくなる展開になっているか？
- [ ] 読み終えた時に「考える力」が向上したと感じられるか？
- [ ] 記事自体が学習体験として機能しているか？

## 重要原則 ⚠️

1. **すべての段階で「think harder」を実践**

   - 対話中の深掘り
   - 構造化時の本質追求
   - 執筆時の読者思考プロセス設計

2. **効率性の罠を避ける**

   - 「分かりやすい」ことを最優先にしない
   - 読者の思考プロセスを奪わない
   - 結論の押し付けを避ける

3. **真の学習体験の創造**
   - 記事内容だけでなく構造そのものが教育
   - 読者の能動的発見を促進
   - 思考の筋肉を鍛える設計

この手法により、単なる情報伝達ではなく、読者の知的成長を促進する記事を生成できる。
